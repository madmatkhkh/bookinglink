# نوبت‌لینک (bookinglink)

پلتفرمِ multi-tenant نوبت‌دهی برای متخصصانِ فردی (مشاور، مربی، کوچ). هر متخصص یک صفحه‌ی برندشده‌ی اختصاصی می‌گیرد (`/{slug}`) و لینکش را در بایوی اینستاگرام می‌گذارد.

> اسمِ «نوبت‌لینک» placeholder است — فقط `PLATFORM_NAME` در `src/lib/config.ts` را عوض کنید.

## استک

Next.js 14 (App Router) · TypeScript · Tailwind CSS · Supabase (Postgres + RLS) · تقویمِ جلالی

## راه‌اندازی

### ۱. دیتابیس (Supabase)

- یک پروژه‌ی جدید در Supabase بسازید.
- محتوای `supabase-schema.sql` را کامل در SQL Editor اجرا کنید.
  - همه‌ی جدول‌ها با RLS فعال ساخته می‌شوند و هیچ policy عمومی ندارند — دسترسی فقط از سمتِ سرور با service role است.

### ۲. متغیرهای محیطی

`.env.example` را به `.env.local` کپی و پر کنید:

| متغیر | توضیح |
|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | آدرسِ پروژه‌ی Supabase |
| `SUPABASE_SERVICE_ROLE_KEY` | کلیدِ service role (فقط سمتِ سرور استفاده می‌شود) |
| `SUPER_SECRET` | رمزِ ورود به پنلِ سوپرادمین (`/super`) — یک رشته‌ی بلندِ تصادفی |
| `AUTH_SECRET` | کلیدِ امضای HMAC کوکی‌های ورود — یک رشته‌ی بلندِ تصادفیِ دیگر |

### ۳. اجرا

```bash
npm install
npm run dev      # توسعه
npx next build   # باید با exit 0 تمام شود
```

### ۴. ساختِ اولین متخصص

1. به `/super` بروید و با `SUPER_SECRET` وارد شوید.
2. یک tenant بسازید: slug (مثلاً `sara`) + موبایلِ متخصص + نامِ نمایشی.
3. متخصص به `/{slug}/panel` می‌رود، با OTPِ موبایلش وارد می‌شود و در تبِ «تنظیمات» پروفایل، سرویس‌ها و ساعاتِ کاری‌اش را پر می‌کند.
4. لینکِ `/{slug}` آماده‌ی گذاشتن در بایو است.

> **نکته‌ی OTP:** تا وقتی سرویسِ پیامک وصل نشده، کدِ OTP در پاسخِ API با کلیدِ `dev_code` برمی‌گردد (فقط برای توسعه). قبل از production واقعی، ارسالِ پیامک را در `api/t/[slug]/otp` و `api/t/[slug]/panel/login` جایگزین کنید و `dev_code` را حذف کنید.

## نقشه‌ی مسیرها

| مسیر | چی |
|---|---|
| `/{slug}` | صفحه‌ی عمومیِ برندشده‌ی متخصص |
| `/{slug}/book/{serviceId}` | فلوی رزرو (تقویم ← OTP ← کارت‌به‌کارت) |
| `/{slug}/my` | پنلِ مراجع (نوبت‌های من، لغو) |
| `/{slug}/panel` | پنلِ متخصص (نوبت‌ها، تقویم، تنظیمات، لینکِ من) |
| `/super` | سوپرادمین (ساخت/تعلیقِ tenant) |

## اصولِ معماری

- **یک codebase، یک دیتابیس** — مشتریِ جدید = یک ردیف در `tenants`.
- **هرچه بینِ دو مشتری فرق می‌کند = دیتا؛ هرچه ثابت است = کد.**
- **اسلات‌ها محاسبه می‌شوند، ذخیره نمی‌شوند:** (الگوی هفتگی + استثناها) − رزروهای زنده. ضامنِ race condition ایندکسِ unique دیتابیس است (`bookings_slot_unique`).
- **اعدادِ ذخیره‌شده همیشه لاتین**، تبدیل به فارسی فقط هنگامِ نمایش.
- **فیچرِ اختصاصیِ مشتری = ماژول** در جدولِ `tenant_features`، هرگز کدِ اختصاصی در هسته.
