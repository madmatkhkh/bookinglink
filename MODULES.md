# MODULES.md — طراحی سیستم ماژولار نوبت‌لینک (قابلیت = محصول)

> سند طراحی — هنوز هیچ کدی از این سند پیاده نشده. این فایل را کنار CLAUDE.md در ریپو نگه دار؛ هر فازی که پیاده شد، وضعیتش را همین‌جا آپدیت کن.

## 1. اصل بنیادی

**«ماژولار در دیتا، یکپارچه در کد»**

- **ماژول = کد واقعی React + API route + جدول** — هرگز UI ماژول را دیتایی/JSON-رندر نمی‌کنیم.
- **فعال‌بودن ماژول = دیتا** — یک ردیف در دیتابیس، قابل روشن/خاموش از سوپرادمین بدون دیپلوی.
- **نیچ = باندل پیش‌فرض ماژول‌ها + برچسب‌ها** — نه یک محصول جدا. ساختن نیچ جدید یعنی انتخاب ماژول از کاتالوگ، نه نوشتن پنل از صفر.

این ادامه‌ی مستقیم معماری سه‌لایه‌ی فعلی است (هسته / تمپلیت نیچ / tenant_features) — فقط لایه‌ی سوم را از چند کلید ad-hoc به یک سیستم رسمی با کاتالوگ، وابستگی، و منبع فعال‌سازی ارتقا می‌دهد.

## 2. اینونتوری کامل — تکلیف تک‌تک قابلیت‌های موجود

برچسب‌ها:
- **[هسته]** — همیشه روشن، قابل خاموش‌کردن نیست، در کاتالوگ نمی‌آید.
- **[پلتفرم]** — ماژول قابل‌فروش/قابل‌سوییچ، برای هر نیچی معنی دارد.
- **[نیچ:psy]** — ماژول مخصوص نیچ روانشناسی (نیچ‌های دیگر ممکن است معادل خودشان را داشته باشند).
- **[ارتقا]** — الان psy است ولی کاندید ارتقا به سطح پلتفرم هنگام ساخت نیچ دوم.

### 2.1 هسته (Core) — بدون کلید، همیشه فعال

| قابلیت | جای فعلی در کد |

| کلید ماژول | چه چیزی را کنترل می‌کند | وضعیت |

## 3. طراحی دیتابیس

هر سه تغییر **additive** است — هیچ drop/rename روی دیتابیس زنده.

### 3.1 جدول جدید `modules` — کاتالوگ محصولات

```sql
create table public.modules (
  key           text primary key,          -- 'waitlist', 'psy_stages', ...
  display_name  text not null,             -- «لیست انتظار»
  description   text not null default '',
  scope         text not null default 'platform',  -- 'platform' | 'psychology' | 'beauty_salon' | ...
  depends_on    jsonb not null default '[]',        -- ['pay_online|pay_card'] — آرایه؛ عضو با | یعنی «حداقل یکی»
  default_on    boolean not null default false,     -- برای tenantهای موجود که ردیف ندارند
  pricing_type  text not null default 'included',   -- 'included' | 'addon_monthly' | 'addon_once'
  price         bigint not null default 0,          -- تومان؛ فقط نمایشی/فاکتور دستی در فاز اول
  is_active     boolean not null default true,      -- خاموش‌کردن ماژول از کل پلتفرم
  sort_order    integer not null default 0
);
```

نکته‌ها:
- `depends_on` با کانونشن `'a|b'` هم «یکی از» را پوشش می‌دهد هم AND ساده — بدون نیاز به موتور قواعد.
- `default_on` مشکل «tenant قدیمی ردیف ندارد» را حل می‌کند: منطق چک = «ردیف tenant اگر هست، همان؛ وگرنه default_on ماژول». دقیقا همان رفتاری که الان PATIENT_FEATURE_KEYS (پیش‌فرض روشن) و multi_therapist (پیش‌فرض خاموش) به‌صورت هاردکد دارند — فقط دیتایی می‌شود.

### 3.2 ارتقای `tenant_features` (نه rename!)

جدول و نامش می‌ماند — چون کد زنده به آن می‌خواند/می‌نویسد. فقط دو ستون:

```sql
alter table public.tenant_features
  add column if not exists source text not null default 'manual',
  -- 'niche_default' | 'addon' | 'trial' | 'manual'
  add column if not exists expires_at timestamptz;  -- فقط برای trial
```

`feature_key` از این به بعد همان `modules.key` است (FK نمی‌گذاریم تا کلیدهای قدیمی نشکنند — اعتبارسنجی در لایه‌ی API).

### 3.3 `niches.default_features` — رسمی‌سازی

ستون از قبل هست (jsonb). فقط قراردادش رسمی می‌شود: **آرایه‌ی کلیدهای ماژول** از کاتالوگ. موقع ساخت tenant، به‌ازای هر کلید یک ردیف `tenant_features` با `source='niche_default'` ساخته می‌شود. صفحه‌ی `/super/niches` به‌جای textarea خام، چک‌لیست از روی کاتالوگ `modules` نشان می‌دهد.

## 4. طراحی سمت کد

### 4.1 مانیفست ماژول — `src/lib/modules.ts`

یک رجیستری تایپ‌شده که **تنها منبع حقیقت** درباره‌ی اینکه هر ماژول چه چیزهایی به پنل اضافه می‌کند:

```ts
type ModuleManifest = {
  key: string
  // تب سایدبار (اگر ماژول تب دارد)
  nav?: { icon: string; label: string; section: 'main' | 'bottom'; badgeSource?: 'waitlist' | 'pending_payments' }
  // زیرتب داخل یک تب موجود (مثل reviews داخل تب رشد)
  subTabOf?: { tab: string; key: string; label: string }
  // بخش تنظیمات (اگر دارد)
  settingsSection?: { group: string; key: string; icon: string; label: string }
  // ویجت/قابلیت در پنل مراجع (فقط فلگ — رندرش داخل کد /my شرطی است)
  clientPanel?: string[]
  // routeهای API که این ماژول گیت می‌کند (مستندسازی + تست)
  apiRoutes: string[]
}
```

- سایدبار (`navItems`/`settingsGroups` که الان هاردکدند) از روی `enabledModules ∩ رجیستری` ساخته می‌شود.
- شرط‌های موجود مثل `showBookingsTab` و `me?.multiTherapist` حالت‌های خاص همین الگو هستند و در آن حل می‌شوند.

### 4.2 گارد API — `requireModule`

یک helper کنار `requirePanelAuth` در `lib/tenant.ts`:

```ts
requireModule(tenantId, 'waitlist')  // خاموش → 403 با پیام فارسی روشن
```

هر route ماژولی (waitlist, reviews, analytics, campaigns, discount-codes, ...) در خط اولش این را صدا می‌زند. **این مهم‌تر از UI است** — مخفی‌کردن تب کافی نیست، چون route بدون گارد یعنی ماژول «فروختنی» در عمل مجانی و باز است.

کش: یک fetch از `tenant_features` per request کافی است (helper همه‌ی ردیف‌های tenant را یک‌جا می‌خواند و Map برمی‌گرداند: `getEnabledModules(tenantId)`)، چون routeها همین حالا هم چند کوئری می‌زنند.

### 4.3 شکستن فایل 5700 خطی — تدریجی و بدون تغییر رفتار

مقصد: `src/app/[slug]/panel/modules/<key>/` — هر تب یک کامپوننت مستقل که state خودش را دارد و فقط چند prop مشترک (slug، me، api helper) می‌گیرد. ترتیب جداسازی از کم‌تنیده‌ترین:

1. `growth` (waitlist/reviews/analytics/campaigns — از قبل زیرتب‌های تمیز دارد و 4 ماژول جدا می‌شود)
2. `finance`
3. `staff`
4. `bookings` (تأیید پرداخت‌ها)
5. `schedule` — تنیده‌تر (agenda به sessions/stages وصل است)
6. `patients` — آخر از همه، بزرگ‌ترین و تنیده‌ترین

قانون: **هر مرحله یک PR/جلسه‌ی جدا، build سبز، رفتار عینا یکسان.** بازنویسی بزرگ یک‌جا ممنوع — محصول زنده است.

## 5. مهاجرت کلیدهای موجود

| کلید قدیمی در دیتابیس | کلید کاتالوگ | راهکار |
|---|---|---|
| `multi_therapist` | `multi_staff` | ردیف کاتالوگ با key قدیمی ساخته می‌شود (`multi_therapist`) — دیتا دست نمی‌خورد؛ فقط display_name عمومی می‌گیرد. تغییر کلید ارزش ریسکش را ندارد. |
| `card_to_card` | همان | ردیف کاتالوگ |
| `patient_buy_extra_session` | همان | ردیف کاتالوگ، `default_on=true` |
| `patient_self_cancel` | همان | ردیف کاتالوگ، `default_on=true` |

یعنی: **کاتالوگ خودش را با واقعیت دیتابیس تطبیق می‌دهد، نه برعکس.** هیچ tenant زنده‌ای چیزی حس نمی‌کند.

نکته‌ی `pay_online`: الان روشن/خاموشش per-resource است (`payment_methods` در پروفایل دکتر). این می‌ماند — ماژول سطح tenant تعیین می‌کند «این قابلیت اصلا در دسترس هست یا نه»، تنظیم per-resource تعیین می‌کند «این دکتر استفاده‌اش می‌کند یا نه». (دو لایه‌ی مستقل — همین الگو الان برای card_to_card برقرار است: سوپرادمین اجازه می‌دهد، دکتر روشن می‌کند.)

## 6. باندل‌ها و قیمت‌گذاری — کنترل انفجار ترکیب‌ها

با ~13 ماژول پلتفرمی، فروش کاملا à la carte یعنی ماتریس تست و پشتیبانی غیرقابل‌مدیریت برای تیم یک‌نفره. پیشنهاد:

- **3 باندل استاندارد** (فقط preset از ماژول‌ها، بدون مکانیک جدید):
  - **پایه:** نوبت‌دهی + یکی از روش‌های پرداخت + فرم رزرو + کنسل مراجع
  - **حرفه‌ای:** پایه + پرداخت آنلاین + یادآور + لیست انتظار + نظرات + آنالیتیکس + کد تخفیف + جلسه‌ی آنلاین
  - **کلینیک:** حرفه‌ای + چندپرسنلی + کمپین‌ها
- **Add-onهای مستقل معدود:** `clinical_notes`، `custom_domain` (وقتی ساخته شد)، شاید `campaigns`.
- فاز اول **بدون اتوماسیون بیلینگ**: سوپرادمین دستی فعال می‌کند (`source='addon'`)، فاکتور بیرون سیستم. ledger موجود برای کمیسیون است و بعدا می‌تواند اشتراک را هم بگیرد — الان نه.
- باندل‌ها هم می‌توانند یک ردیف در همان جدول `modules` با `scope='bundle'` باشند یا فقط یک ثابت در کد سوپرادمین — پیشنهاد: **ثابت در کد** فعلا؛ جدول وقتی لازم می‌شود که بیلینگ خودکار بیاید.

## 8. تصمیم‌های رد شده (که دوباره وسوسه نشویم)

- ❌ **UI دیتایی/JSON-driven برای ماژول‌ها** — فرم‌بیلدر استثنا بود چون هر دکتر فرم متفاوت می‌خواهد؛ تب مالی برای همه یکی است.
- ❌ **Rename جدول tenant_features یا کلیدهای موجود** — ریسک روی دیتابیس زنده، سود صفر.
- ❌ **موتور قواعد وابستگی پیچیده** — آرایه‌ی `depends_on` با کانونشن `|` تا اطلاع ثانوی کافی است.
- ❌ **بیلینگ/اشتراک خودکار در فاز اول** — سوپرادمین دستی + فاکتور بیرونی.
- ❌ **Plugin loader / مارکت‌پلیس / lazy-load ماژول‌ها** — یک monolith با import ثابت؛ tree از build خارج نمی‌شود و لازم هم نیست.
- ❌ **بازنویسی یک‌جای PsychologyAdmin** — فقط جداسازی تدریجی تب‌به‌تب.
