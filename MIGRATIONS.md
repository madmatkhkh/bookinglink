# تغییر اسکیما

## وضعیت فعلی

`schema.sql` یک خروجی pg_dump یکپارچه از اجرای کامل مهاجرت‌های `0001..0054`
است. پوشه‌ی `migrations/` دیگر در ریپو نیست — تاریخچه در همان یک فایل تثبیت شده.

⚠️ **`schema.sql` با `drop schema public cascade` شروع می‌شود.** یعنی فقط برای
دیتابیس تازه/خالی است. هرگز روی دیتابیسی که داده‌ی واقعی دارد اجرایش نکن.

runner خودکار (`scripts/migrate.mjs`) هم در ریپو نیست — عمدا حذف شد. یعنی
اعمال تغییر روی دیتابیس زنده **دستی** است.

## دیتابیس تازه (نصب از صفر)

کل `schema.sql` را در Supabase → SQL Editor پیست کن و Run بزن. تمام.

## دیتابیس زنده (تغییر اسکیما)

1. یک فایل تازه بساز: `migrations/0055_<توضیح_کوتاه>.sql`
   (شماره از آخرین شماره‌ی ذکرشده در هدر `schema.sql` +۱)
2. قواعد اجباری:
   - **فقط additive** — `add column if not exists`، `create table if not exists`،
     `create index if not exists`. هیچ `drop`/`rename` روی ستون یا جدول زنده.
     (بازسازی مخرب از migration 0019 رسما منسوخ است.)
   - **idempotent** — اجرای دوباره نباید خطا بدهد یا دیتا را خراب کند.
     برای seed از `on conflict do nothing` استفاده کن.
   - مقادیر پیش‌فرض ستون‌ها هم مشمول قانون «رقم لاتین» و «بدون اعراب» هستند.
3. محتوای همان فایل را در SQL Editor اجرا کن.
4. همان SQL را به انتهای `schema.sql` هم append کن و شماره‌ی هدرش را بامپ کن،
   تا نصب از صفر و دیتابیس زنده از هم جدا نیفتند.

## ترتیب امن استقرار

کد و مهاجرت باید از **هر دو جهت** امن باشند، چون دیپلوی ورسل و اجرای SQL
هم‌زمان نیستند:

- **کد اول برود:** ستون/جدول تازه هنوز نیست → کد باید fail-open باشد
  (مثل روت invoices که نبودن جدول را لیست خالی می‌گیرد، نه خطا).
- **مهاجرت اول برود:** ستون تازه هست ولی هیچ‌کس نمی‌نویسدش → بی‌اثر.

الگوی جاافتاده‌ی پروژه: قابلیت تازه پشت یک کلید در `platform_settings` یا
`tenant_features` باشد که تا آگاهانه روشن نشود، رفتار عینا قبلی می‌ماند.

## سوییچ‌های آگاهانه‌ی موجود

دو کلید در `platform_settings` که عمدا دست‌نخورده مانده‌اند:

| کلید | مقدار فعلی | معنی |
|---|---|---|
| `plans_enforced` | `true` | preset پلن‌ها روی ماژول‌ها اعمال می‌شود |
| `plan_prices.enabled` | `false` | **بیلینگ خاموش** — هیچ‌کس حق اشتراک شارژ نمی‌شود |

روشن‌کردن بیلینگ:

```sql
update platform_settings
set value = jsonb_set(value, '{enabled}', 'true')
where key = 'plan_prices';
```

قبل از این کار: چرخه‌ی تست کامل با یک tenant تستی (پرداخت شبیه‌سازی‌شده →
کارمزد → cron اشتراک → تسویه‌ی net → فاکتور ماه بعد) و اطلاع‌رسانی به مشتری‌ها.
