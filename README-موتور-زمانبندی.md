## باگی که رفع شد

`slots.ts` مدت سرویس را فقط هنگام *ساختن* اسلات به کار می‌برد. هنگام
*تشخیص تداخل* فقط ساعت شروع را مقایسه می‌کرد.

یعنی: رنگ مو (120 دقیقه) ساعت 10:00 رزرو می‌شد، و سیستم 10:30 را برای ناخن
(30 دقیقه) خالی می‌دید و می‌فروخت.

`bookings_slot_unique` هم جلویش را نمی‌گرفت، چون روی ساعت شروع دقیق است و
10:30 با 10:00 دو ردیف متفاوت‌اند. تداخل دو بازه را هیچ ایندکس یکتایی
نمی‌تواند بیان کند — راه‌حل استاندارد پستگرس exclusion constraint است.

### تست الگوریتم

روز کاری 9:00 تا 13:00، رنگ موی رزروشده 10:00 تا 12:00:

| سرویس | قبل | بعد |
|---|---|---|
| ناخن (30) | 9:00 9:30 **10:30 11:00 11:30** 12:00 12:30 | 9:00 9:30 12:00 12:30 |
| یک‌ساعته (60) | 9:00 **11:00** 12:00 | 9:00 12:00 |
| رنگ مو (120) | **9:00 11:00** | (هیچ‌کدام) |

پررنگ‌ها اسلات‌های متداخلی هستند که قبلا فروخته می‌شدند.

`npx next build` → سبز. `npx tsc --noEmit` → صفر خطا در `src/`.

## محدودیت شناخته‌شده (باگ نیست، تصمیم محصولی)

شبکه‌ی اسلات‌ها از ابتدای روز با گام `duration_minutes` ساخته می‌شود. یعنی
شکافی که از یک سرویس کوچک‌تر باشد قابل‌فروش نیست. برای سالن که پشت‌سرهم
بودن نوبت‌ها مستقیما درآمد است، احتمالا باید به گام ثابت (مثلا 15 دقیقه‌ای)
تغییر کند. عمدا الان دست نزدم — تصمیم توست، و باگ فعلی را نمی‌ساخت.

## نکته درباره‌ی روانشناسی

روانشناسی هم همین باگ را به‌شکل نهفته دارد: `duration_online: 50` فقط
نمایشی است و موتور نمی‌بیندش. اگر دکتری 9:00 و 9:30 منتشر کند با جلسه‌ی
50 دقیقه‌ای، سیستم هر دو را می‌فروشد. الان امن است چون دکترها دستی فاصله
می‌گذارند — «امن به‌قرارداد، نه به‌ساختار».

عمدا دست نزدم: روانشناسی زنده است و داده‌ی واقعی دارد. اگر خواستی، همین
مدل بازه‌ای برایش هم قابل‌اعمال است، ولی جلسه‌ی خودش را می‌خواهد.
